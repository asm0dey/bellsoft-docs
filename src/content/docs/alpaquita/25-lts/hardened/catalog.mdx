## Available hardened repositories

BellSoft provides hardened container images for multiple programming languages and runtimes, all based on Alpaquita Linux. The same repositories are available across all container registries (Docker Hub, GitHub CR, Microsoft CR, Google CR, and Amazon ECR).

| Repository name | Description | Supported versions |
|-----------------|-------------|--------------------|
| `hardened-base` | Minimal hardened base images for running pre-built applications | Alpaquita Linux Stream |
| `hardened-liberica-runtime-container` | Hardened images with Liberica JDK Lite (OpenJDK) | LTS versions (21 and 25) and the latest non-LTS release |
| `hardened-liberica-native-image-kit-container` | Hardened images with Liberica Native Image Kit (GraalVM) | NIK 23 (JDK 17, 21) and NIK 25 (JDK 25) |
| `hardened-python` | Hardened images with Python and basic Python utilities | Python 3.x |
| `hardened-nodejs` | Hardened images with Node.js | Latest LTS version |
| `hardened-go` | Hardened Alpaquita Linux images for building Go applications | Go 1.x |
| `hardened-gcc` | Hardened Alpaquita Linux images with the GCC compiler, tools, and libraries for development in C/C++ | GCC 15.x |

<Aside type="note">
The **standard**, **nonroot**, and **distroless** variants of each image are hosted in the same repository, distinguished by tag. For the differences between them, see [Hardened image types](/bellsoft-docs/alpaquita/25-lts/hardened/overview/#hardened-image-types).
</Aside>

## Image tags

Build a tag for the OS-based hardened images and check it against what is
actually published:

The per-language details and examples follow. (Liberica JDK and Liberica NIK
hardened images have their own pickers — see the linked pages below.)

<Tabs syncKey="lang">
  <TabItem label="Base">

The repository contains hardened images with a minimal set of OS components for running statically linked programs or programs that depend only on the standard C library.

The available image tags are defined by the following pattern:

`[nonroot-|distroless-]<libc>`

where

- **nonroot** and **distroless** are [types of images](/bellsoft-docs/alpaquita/25-lts/hardened/overview/#hardened-image-types)
- `<libc>` specifies the version of the C library and can take the following values:
  - glibc
  - musl

<details>
<summary>Tag examples</summary>

| Tag | Description |
|-----|-------------|
| `glibc` | standard image with glibc |
| `musl` | standard image with musl |
| `nonroot-glibc` | nonroot image with glibc |
| `distroless-musl` | distroless image with musl |

</details>

  </TabItem>
  <TabItem label="GCC">

The repository contains hardened images with tools for C/C++ development based on GCC.

The available image tags are defined by the following pattern:

`<gcc_x.y_version>[-nonroot|-distroless]-<libc>`

where

- `<gcc_x.y_version>` is the version of the gcc, such as 15.2.
- **nonroot** and **distroless** are [types of images](/bellsoft-docs/alpaquita/25-lts/hardened/overview/#hardened-image-types)
- `<libc>` specifies the version of the C library and can take the following values:
  - glibc
  - musl

<details>
<summary>Tag examples</summary>

| Tag | Description |
|-----|-------------|
| `15.2-glibc` | standard image, GCC 15.2 with glibc |
| `15.2-musl` | standard image, GCC 15.2 with musl |
| `15.2-nonroot-musl` | nonroot image, GCC 15.2 with musl |
| `15.2-distroless-glibc` | distroless image, GCC 15.2 with glibc |

</details>

The next code example demonstrates a Dockerfile with an application built using a nonroot image followed by running in a basic distroless image:

```dockerfile
FROM bellsoft/hardened-gcc:15.2-nonroot-musl AS builder

COPY <<'EOF' hello_bellsoft.c
#include <stdio.h>

int main(int argc, char *argv[]) {
    printf("Hello BellSoft!\n");
}
EOF

RUN gcc -Wall -o hello-bellsoft hello_bellsoft.c

FROM bellsoft/hardened-base:distroless-musl

COPY --from=builder /home/appuser/hello-bellsoft .

ENTRYPOINT [ "/home/appuser/hello-bellsoft" ]
```

  <TabItem label="Go">

The repository contains hardened images with the Go compiler. These images are designed for building and debugging programs written in Go.

Only images with the Go version supported by the Go language developers ([https://go.dev](https://go.dev)) are provided.

The available image tags are defined by the following pattern:

`<go_version>[-nonroot|-distroless]-<libc>`

where

- `<go_version>` is the Go version in 1.X or 1.X.Y format, such as 1.25 or 1.25.7.
- **nonroot** and **distroless** are [types of images](/bellsoft-docs/alpaquita/25-lts/hardened/overview/#hardened-image-types)
- `<libc>` specifies the version of the C library and can take the following values:
  - glibc
  - musl

<details>
<summary>Tag examples</summary>

| Tag | Description |
|-----|-------------|
| `1.25-glibc` | standard image, Go 1.25 with glibc |
| `1.25-musl` | standard image, Go 1.25 with musl |
| `nonroot-musl` | nonroot image, latest Go with musl |
| `distroless-glibc` | distroless image, latest Go with glibc |

</details>

The next code example demonstrates a Dockerfile with a Go application built using a nonroot image followed by running in a basic distroless image:

```dockerfile
FROM bellsoft/hardened-go:nonroot-musl AS builder

COPY <<'EOF' hello_bellsoft.go
package main

import "fmt"

func main() {
    fmt.Println("Hello BellSoft")
}
EOF

RUN go build -o hello-bellsoft hello_bellsoft.go

FROM bellsoft/hardened-base:distroless-musl

COPY --from=builder /home/appuser/hello-bellsoft .

ENTRYPOINT [ "/home/appuser/hello-bellsoft" ]
```

  <TabItem label="Node.js">

The repository contains hardened images with Node.js and related utilities (npm).

Only images with the Node.js version supported by the Node.js developers ([https://nodejs.org](https://nodejs.org)) are provided.

The available image tags are defined by the following pattern:

`<nodejs_version>[-nonroot|-distroless]-<libc>`

where

- `<nodejs_version>` is the Node.js version in X, X.Y, or X.Y.Z format, such as 24, 24.13 or 24.13.0.
- **nonroot** and **distroless** are [types of images](/bellsoft-docs/alpaquita/25-lts/hardened/overview/#hardened-image-types)
- `<libc>` specifies the version of the C library and can take the following values:
  - glibc
  - musl

<details>
<summary>Tag examples</summary>

| Tag | Description |
|-----|-------------|
| `24-glibc` | standard image, Node.js 24 with glibc |
| `24-musl` | standard image, Node.js 24 with musl |
| `24-nonroot-glibc` | nonroot image, Node.js 24 with glibc |
| `distroless-musl` | distroless image, latest Node.js with musl |

</details>

The next code example demonstrates a Dockerfile that runs an application in a distroless image:

```dockerfile
FROM bellsoft/hardened-nodejs:distroless-musl

COPY <<'EOF' hello_bellsoft.js
import { createServer } from 'node:http';

const server = createServer((req, res) => {
  res.writeHead(200, { 'Content-Type': 'text/plain' });
  res.end('Hello BellSoft!\n');
});

server.listen(5000, '0.0.0.0', () => {
  console.log('Listening on 0.0.0.0:5000');
});
EOF

ENTRYPOINT [ "/usr/sbin/tini", "--"]
CMD [ "node", "hello_bellsoft.js"]
```

- `<libc>` specifies the version of the C library and can take the following values:
  - glibc
  - musl

<details>
<summary>Tag examples</summary>

| Tag | Description |
|-----|-------------|
| `3.14-glibc` | standard image, Python 3.14 with glibc |
| `3.14-musl` | standard image, Python 3.14 with musl |


## Liberica JDK and Liberica NIK images

The hardened Liberica JDK runtime (`hardened-liberica-runtime-container`) and hardened Liberica Native Image Kit (`hardened-liberica-native-image-kit-container`) repositories have their own documentation:

- Hardened Liberica JDK runtime images are documented in [Liberica JDK container images](/bellsoft-docs/liberica-jdk/containers/).
- Hardened Liberica NIK images are documented in [Liberica NIK container images](/bellsoft-docs/liberica-nik/containers/).
