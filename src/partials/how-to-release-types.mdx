## Introduction

OpenJDK security and maintenance updates follow a quarterly schedule, with new releases delivered in January, April, July, and October. Within each quarter, OpenJDK (and therefore Liberica JDK) is published in two flavors: a **Critical Patch Update (CPU)** and a **Patch Set Update (PSU)**.

Both flavors ship at the same time and contain the same security content for that quarter. The difference is in what additional changes they carry. This page explains the distinction so that you can choose the right release for your environment.

Liberica JDK tracks the OpenJDK quarterly cadence and provides both CPU and PSU builds for the supported feature releases.

## Critical Patch Update (CPU)

A **Critical Patch Update** contains only the critical security fixes for the quarter plus a minimal set of changes required to deliver them safely since the previous quarterly release.

Because a CPU keeps the change surface as small as possible, it carries the lowest risk of regressions while still applying every security fix for the quarter. It is the recommended default for production deployments: you get the fully tested security patch without absorbing unrelated functional changes.

The CPU is the build most users should install. Its version is reflected in the standard quarterly update number.

## Patch Set Update (PSU)

A **Patch Set Update** contains everything in the CPU for the same quarter **plus** additional non-critical bug fixes and enhancements.

Choose a PSU when you specifically need one of those extra fixes — for example, a non-security bug fix that addresses an issue you are hitting and that is not included in the CPU. Because a PSU carries more changes, it has a slightly larger surface for behavioral differences than the CPU.

PSUs and CPUs of the same quarter share identical security content, so you never trade security coverage by picking one over the other.

<Aside type="note">
The PSU of a given quarter becomes the base for the next quarter's CPU. Non-critical fixes that first appear in a PSU are therefore rolled into the following quarterly CPU.
</Aside>

## Which one should you choose?

As a rule of thumb, install the **CPU** unless you need a fix that is only available in the **PSU**. The CPU gives you the full quarterly security update with the smallest amount of additional change, which is what most production systems want.

Switch to the PSU only when you have identified a specific non-critical fix or enhancement in it that you require.

## CPU vs PSU at a glance

| Aspect | CPU (Critical Patch Update) | PSU (Patch Set Update) |
|---|---|---|
| Contents | Critical security fixes only, plus a minimal set of supporting changes | Everything in the CPU **plus** additional non-critical bug fixes and enhancements |
| Security content | Full quarterly security fixes | Full quarterly security fixes (identical to the CPU) |
| Change surface | Smallest | Larger |
| Regression risk | Lowest | Slightly higher |
| When to use | Default choice for production | When you need a specific non-critical fix or enhancement only available here |
| Relationship | Standard quarterly update build | Becomes the base for the next quarter's CPU |

## Identifying CPU and PSU builds

You can tell CPU and PSU builds apart by their version string and by how they are presented in the download center:

- The version string identifies the feature release and quarterly update (for example, the third update of a feature release). CPU and PSU builds of the same quarter are published as distinct builds, and the build metadata distinguishes them.
- The [Liberica JDK Download Center](https://bell-sw.com/pages/downloads/) lets you select the build you need for your platform and architecture.

For the exact contents of a given build, see the [release notes](../../release-notes/) for that version.
